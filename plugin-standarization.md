# Master Orchestrator — Cross-Platform Compiler & Module Assembly Standard

## Why this changed

Our original strategy kept sub-plugins as separate files (`font-colors-main.js`) sitting directly inside the vault plugins directory. While functional on local desktop systems, this architecture introduced critical cross-platform failures because the Obsidian Sync cloud pipeline filters and ignores unrecognized custom JavaScript or JSON file naming patterns. It *only* syncs `main.js`, `manifest.json`, and `styles.css`.

### The Automated Fix
We combine the seamless cross-platform delivery of a single compiled file with an autonomous asset pipeline: **Compile-Time Module Assembly**. 

You write sub-plugins as cleanly separated standalone TypeScript modules within the `src/modules/` directory. During compilation, a build script automatically generates a central static registry file at `src/modules.ts`, compiling everything into a highly optimized, production-ready `main.js`. You **never** modify `main.ts` to register a new sub-plugin.


## 2. Dynamic Bundle Automation & The Module Registry (`src/modules.ts`)

The compiler script automatically scans the `src/modules/` subdirectory and outputs a hard structural mapping file to `src/modules.ts`. This file acts as the source-of-truth registry matching your configuration profile keys.

### String-Transformation Compilation Engine Rules

The compilation engine derives runtime asset wrappers from raw kebab-case file inputs using a strict three-tier conversion string pipeline:

1. **Module Registry ID:** Kebab-case filenames are preserved as raw **kebab-case** string keys inside the `autoModules` lookup dictionary (e.g., `bases-kanban.ts` $\rightarrow$ `'bases-kanban'`). This ensures an exact 1:1 match with your `data.json` profile keys.
2. **Class References:** Kebab-case filenames map to an matching exported PascalCase class identifier (e.g., `vault-size-bar.ts` $\rightarrow$ `VaultSizeBarModule`).
3. **Fallback Profiles:** Kebab-case filenames map to an matching exported UPPER_SNAKE_CASE configurations dictionary signature (e.g., `notion-like-icons.ts` $\rightarrow$ `NOTION_ICONS_DEFAULTS`).

### The Compiled Registry Interface File (`src/modules.ts`)

The auto-generated mapping contract file must be generated with this explicit format:

```typescript
// Auto-generated module exports
import { CryptomatorKeepAliveModule, CRYPTOMATOR_KEEP_ALIVE_DEFAULTS } from './modules/cryptomator-keep-alive';
import { BasesKanbanModule, BASES_KANBAN_DEFAULTS } from './modules/bases-kanban';

interface ModuleMetadata {
    classRef: any;
    defaults: any;
}

export const autoModules: Record<string, ModuleMetadata> = {
    'cryptomator-keep-alive': { classRef: CryptomatorKeepAliveModule, defaults: CRYPTOMATOR_KEEP_ALIVE_DEFAULTS },
    'bases-kanban': { classRef: BasesKanbanModule, defaults: BASES_KANBAN_DEFAULTS }
};

```

## 3. Sub-Module Export & Lifecycle Contract Update

**Context:** Every module behaves as an isolated workflow engine communicating directly with the root context database.

* **Module Registration:** Every class dropped into `src/modules/` must follow the compiled name configuration mapping precisely (e.g., `class VaultSizeBarModule extends Component` inside `vault-size-bar.ts`).
* **Constructor Type Contract:** Every module class constructor must implement the uniform 4-argument dependency injection signature required by the orchestrator initialization routine:

```typescript
constructor(app: App, manifest: PluginManifest, pluginInstance: any, moduleId: string) {
    super();
    this.app = app;
    this.manifest = manifest;
    this.pluginInstance = pluginInstance;
    this.moduleId = moduleId;
}

```

* **Data Pathing:** Sub-modules are strictly forbidden from writing standalone configuration JSON sheets. They must read and write from their dedicated slice inside the orchestrator's centralized settings object via an库 module token string matching the kebab-case key:

```typescript
get settings(): MySettingsInterface {
    return this.pluginInstance.settings.modulesData[this.moduleId];
}

```

* **Setting Tabs:** The signature for rendering interfaces must match `renderSettings(containerEl: HTMLElement)` so the master orchestrator can delegate layout elements blindly during loop executions.

## 4. Master Orchestrator Overhaul (`src/main.ts`)

* **Type Safety:** The orchestrator must declare its root variable structure using `declare settings: OrchestratorSettings;` to cleanly bypass internal property naming collisions inside Obsidian's inheritance chain.
* **UI Title Parsing Resilience:** The master setting tab formats its display titles directly from the generated kebab-case module IDs. The layout configuration tab converts kebab-case tokens to readable words (e.g., `bases-kanban` $\rightarrow$ `Bases Kanban`), ensuring lookups are completely automated.
* **Initialization Loop:** The bootloader must import `autoModules` from the generated `./modules` map, assign incoming defaults map profiles dynamically to `settings.modulesData`, and instantiate enabled items using an active key array loop:

```typescript
Object.keys(autoModules).forEach((id) => {
    if (this.settings.enabledModules[id]) {
        this.startModule(id);
    }
});

```

## 5. Development Checklist Revamping

When introducing a new sub-plugin tool, execute these steps without exception to ensure it compiles safely:

1. **Create the file:** Drop a new kebab-case `.ts` file into `src/modules/` (e.g., `custom-highlighter.ts`).
2. **Build the Class:** Extend Obsidian's `Component` class, matching the expected PascalCase structure: `CustomHighlighterModule`.
3. **Export Defaults:** Export a settings configuration contract translated to uppercase snake-case: `CUSTOM_HIGHLIGHTER_DEFAULTS`.
4. **Compile:** Run `npm run build`. The compiler will automatically generate the updated import mapping strings inside `src/modules.ts`, bind the matching key (`'custom-highlighter'`), and bundle the addition directly into your distribution `main.js`.

## 6. Unified Layout Styling Integration Contract

### The Distribution Failure Rule

In our legacy architecture, sub-plugins carried separate cascading stylesheets (`notion-like-icons-styles.css`, `vault-size-bar.styles.css`). Because the Obsidian Sync architecture actively filters and drops custom-named CSS sheets inside community plugin directories, **individual style sheets are strictly banned from active runtime environments**. If split assets are deployed, sub-modules will experience critical visual breakdowns (distorted modal windows, broken grid scaling, unstyled sidebars, or layout overflow anomalies).

### Mandatory Consolidated Compilation

Every sub-module must aggregate its presentation layers into a singular, production-ready stylesheet located exactly at the project root folder directory:

```text
notion-like-plugins/styles.css

```

Obsidian reads this single file automatically alongside the compiled `main.js` binary block on boot, allowing full style registration across desktop and mobile devices.

### Module-Specific Layout Signatures

When adding styling configurations to the master root `styles.css` file, divide rules cleanly using standardized commented modules header sections to prevent collision drift across the workspace:

```css
/* ==========================================================================
   MODULE STYLES: [moduleId-token]
   ========================================================================== */
.obsidian-module-specific-class {
    /* Scoped layout attributes here */
}

```

### Build Phase Validation Checklist

Before finalizing a feature deployment, execute these production checks:

1. Verify no standalone `.css` documents remain in the `src/modules/` or vault folder zones.
2. Verify all UI element identifiers, font parameters, grid matrices, or modal wrappers are fully merged into the singular root `styles.css` manifest.
3. Run `Reload app without saving` via the Command Palette to flush the application's viewport layer and force-reload the unified styles.
