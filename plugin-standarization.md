## 6. Unified Layout Styling Integration Contract

### The Distribution Failure Rule
In our legacy architecture, sub-plugins carried separate cascading stylesheets (`notion-like-icons-styles.css`, `vault-size-bar.styles.css`). Because the Obsidian Sync architecture actively filters and drops custom-named CSS sheets inside community plugin directories, **individual style sheets are strictly banned from active runtime environments**. If split assets are deployed, sub-modules will experience critical visual breakdowns (distorted modal windows, broken grid scaling, unstyled sidebars, or layout overflow anomalies).

### Mandatory Consolidated Compilation
Every sub-module must aggregate its presentation layers into a singular, production-ready stylesheet located exactly at the project root folder directory:
```text
notion-like-plugins/styles.css

```

Obsidian reads this single file automatically alongside the compiled `main.js` binary block on boot, allowing full style registration across desktop and mobile devices.

### Module-Specific Layout Signatures

When adding styling configurations to the master root `styles.css` file, divide rules cleanly using standardized commented modules header sections to prevent collision drift across the workspace:

```css
/* ==========================================================================
   MODULE STYLES: [moduleId-token]
   ========================================================================== */
.obsidian-module-specific-class {
    /* Scoped layout attributes here */
}

```

### Build Phase Validation Checklist

Before finalizing a feature deployment, execute these production checks:

1. Verify no standalone `.css` documents remain in the `src/modules/` or vault folder zones.
2. Verify all UI element identifiers, font parameters, grid matrices, or modal wrappers are fully merged into the singular root `styles.css` manifest.
3. Run `Reload app without saving` via the Command Palette to flush the application's viewport layer and force-reload the unified styles.
